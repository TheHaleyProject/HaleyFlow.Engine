HaleyFlow Protocol

A language-agnostic specification for describing workflow state machines and their execution policies. Any interpreter, engine, or runtime that implements this protocol will behave identically given the same definition and policy JSON — regardless of language or platform.

Overview

HaleyFlow Protocol defines two complementary JSON documents that together describe a complete workflow:

Document Describes Changes?
Definition JSON The structural skeleton — states and the events that move between them. Versioned. Immutable once an instance is created against it.
Policy JSON The behavioural overlay — what hooks fire upon entering a state, in what order, with what parameters, and how completion is determined. Versioned. Locked to an instance at creation time.
Protocol vs Implementation This document describes the protocol — the contract between the JSON author and any compliant interpreter. It does not describe any specific engine or runtime. Any implementation (in any language) that correctly interprets these documents is a conforming HaleyFlow interpreter.

Core Concepts

State

A named position in the lifecycle of an entity. An entity (e.g. a loan application, a change request) is always in exactly one state at any point in time. States are stable — an entity remains in a state until an event causes a transition.

Event

A numeric code that triggers a transition. Events are the only mechanism for moving between states. The same event code may be used from different states — it routes to different transitions depending on the current state of the entity.

Transition

A directed edge from one state to another, identified by a from state, a to state, and an event code. When an event fires while an entity is in from, the entity moves to to.

Hook

A side effect that fires upon arrival at a state. Hooks are identified by a route string — a stable identifier that a handler can subscribe to. Hooks may be gate or effect, and may carry parameters from the policy.

Policy

A behavioural overlay on top of a definition. The policy specifies what happens when a state is entered — which hooks fire, in what order, with what parameters, and how completion is determined. A definition is valid without a policy; a policy without a definition is meaningless.

Definition JSON — States

States are declared in a top-level states array. Each state has a name and optional flags. 

"states": [
  { "name": "ApplicationReceived", "is_initial": true },
  { "name": "KycCheck" },
  { "name": "CreditAssessment" },
  { "name": "ManagerApproval" },
  { "name": "Approved",  "is_final": true },
  { "name": "Rejected",  "is_final": true }
]
FieldTypeRequiredDescription
namestringrequiredUnique name for this state. Case-insensitive matching is used by interpreters.
is_initialbooleanoptionalMarks the entry point. Exactly one state must be initial.
is_finalbooleanoptionalMarks a terminal state. An entity in a terminal state cannot transition further. Multiple terminal states are allowed.
categorystringoptionalInformational grouping label. Not used by the interpreter.
Rule Exactly one state must have is_initial: true. Zero or more than one initial states is a structural error — the interpreter cannot determine the starting point.

Definition JSON — Transitions

Transitions define valid movements between states. Each transition is identified by the combination of from state and event code — this pair must be unique. 

"transitions": [
  { "event": 1001, "from": "ApplicationReceived", "to": "KycCheck" },
  { "event": 2001, "from": "KycCheck",            "to": "CreditAssessment" },
  { "event": 2002, "from": "KycCheck",            "to": "Rejected" },
  { "event": 2007, "from": "ManagerApproval",    "to": "Approved" },
  { "event": 2008, "from": "ManagerApproval",    "to": "Rejected" },
  { "event": 2009, "from": "ManagerApproval",    "to": "ManagerApproval" }
]
FieldTypeRequiredDescription
eventintegerrequiredNumeric event code. Must be unique per from state.
fromstringrequiredThe state the entity must be in for this transition to fire.
tostringrequiredThe state the entity moves to after the transition fires.
eventNamestringoptionalHuman-readable label. For display only — do not use in logic.
Self-loops A transition where from == to is valid and intentional — commonly used for timeout pings or reminder events that re-enter the same state to re-trigger hooks. However, a state with only self-loop transitions and no exit is a structural error.

Policy JSON — Parameter Catalog

The top-level params array defines a catalog of named parameter entries. Each entry has a unique code and a free-form data object. Rules and hooks reference these entries by code — the interpreter resolves the full data and makes it available to handlers at runtime. 

"params": [
  {
    "code": "PARAMS.LOAN.MANAGER.APPROVAL",
    "data": {
      "approval": {
        "rule": "any",
        "selector": {
          "mode": "role",
          "value": "loan_manager"
        }
      }
    }
  }
]

The data payload is intentionally free-form. Interpreters pass it to handlers unchanged. Common uses: approval rules (any/all), role/user selectors, numeric thresholds, feature flags.

Policy JSON — Rules

The rules array defines what happens when an entity arrives at a state. Each rule targets a state by name. When the interpreter moves an entity into that state, it evaluates all matching rules and fires their hooks. 

"rules": [
  {
    "state":  "ManagerApproval",
    "params": [ "PARAMS.LOAN.MANAGER.APPROVAL" ],
    "complete": { "success": 2007, "failure": 2008 },
    "emit": [ ... ]
  }
]
FieldTypeRequiredDescription
statestringrequiredThe state being arrived at (ToState). The rule fires on entry to this state.
viaintegeroptionalIf present, the rule only applies when the state is entered via this specific event code. See The via Field.
paramsstring[]optionalList of param codes from the catalog. Inherited by all hooks in this rule unless a hook defines its own params.
completeobjectoptionalAuto-advance event codes for the transition handler. success and failure are both optional integers.
emitHook[]optionalOrdered list of hooks to fire upon arrival at this state.

Policy JSON — The via Field

The same state can be reached through different events. The via field allows a rule to target only a specific arrival path.

Consider ManagerApproval — it can be reached via event 2005 (a normal RiskReview approval) or via event 2009 (a self-loop timeout ping that re-enters the same state). These two arrivals need completely different hooks. The via field lets you write a targeted rule for one specific path and a general fallback rule for all others: 

// Rule 1 (via-specific): fires ONLY when ManagerApproval is entered via event 2009
{
  "state": "ManagerApproval",
  "via":   2009,
  "emit": [
    { "route": "APP.LOAN.MANAGER.REMINDER",       "type": "effect" },
    { "route": "APP.LOAN.MANAGER.REMINDER.EMAIL", "type": "effect" }
  ]
},
// Rule 2 (general fallback): fires for arrivals NOT claimed by a via-specific rule (e.g. via 2005)
{
  "state":  "ManagerApproval",
  "params": [ "PARAMS.LOAN.MANAGER.APPROVAL" ],
  "emit": [
    { "route": "APP.LOAN.MANAGER.DECISION", "type": "gate", "order": 1,
      "complete": { "success": 2007, "failure": 2008 } }
  ]
}
Priority Rule — via-specific beats general Each transition is governed by exactly one rule. The interpreter uses a two-pass strategy:

Pass 1 — via-specific rules first. All rules that have a via field are evaluated. If a rule's state and via both match the arriving transition, that transition is claimed and no further rules are considered for it.

Pass 2 — general rules as fallback. Rules with no via field are then evaluated, but only for transitions not already claimed in Pass 1.

In the example above: the transition arriving via 2009 is claimed by Rule 1 (the reminder rule). The transition arriving via 2005 is not claimed, so it falls through to Rule 2 (the decision rule).
One rule per transition A transition (identified by FromState + ToState + EventCode) will be governed by at most one rule. Do not write two via-specific rules for the same (state, via) pair — the last one in the array wins and the first is silently ignored.

Policy JSON — Hooks (emit)

The emit array defines the hooks that fire when a rule's state is entered. Each hook is identified by a route — a stable string that an external handler subscribes to. 

{
  "route":    "APP.LOAN.KYC.CHECK",
  "label":    "KYC Check",
  "type": "gate",
  "group":    "kyc_decision",
  "order":    1,
  "params":   [ "PARAMS.LOAN.KYC.OVERRIDE" ],
  "complete": { "success": 2001, "failure": 2002 }
}
FieldTypeRequiredDescription
routestringrequiredUnique stable identifier. Handlers subscribe to this route. Use dot-notation by convention.
labelstringoptionalHuman-readable name. For display and logging only.
typestringoptionalAllowed values: "gate" or "effect". Default: "gate".
orderintegeroptionalExecution sequence within this rule. Lower numbers run first. Hooks without an order run last.
groupstringoptionalLogical grouping label. Informational — used by some interpreters for ACK group tracking.
paramsstring[]optionalHook-own param codes. If absent, inherits parent rule's params.
completeobjectoptionalAuto-advance codes. Only respected on gate hooks. Ignored on effect hooks.

Gate vs Effect

Gate hooks ("type": "gate") participate in the execution chain. Their result can stop the chain, trigger a failure event, or capture a success event. Complete codes on gate hooks are respected.

Effect hooks ("type": "effect") are fire-and-forget. Their result is completely ignored by the interpreter. They cannot carry complete codes — any complete block on an effect hook is silently discarded.

Important If an effect hook defines complete codes, a compliant interpreter must ignore them. A validator should warn the author that these codes will have no effect.

Hook Ordering

Hooks within a rule execute in ascending order value. If two hooks share the same order value, execution order between them is implementation-defined — avoid it (a validator should warn or error if both carry complete codes).

Hooks without an explicit order are assigned the lowest priority — they run after all explicitly ordered hooks complete. This means: the moment you assign any order to any hook in a rule, all un-ordered hooks are pushed to the end.

Mixed gate + effect hooks at the same order value

A gate hook and an effect hook can share the same order value. Both are dispatched together in the same batch. However, ACK completion is asymmetric:

  • The gate hook must ACK before the order is considered resolved. No advancement happens until all gate hooks at this order reach a terminal ACK status.
  • The effect hook can ACK at any time — before or after the gate. An effect ACK on its own does not advance the order. Advancement only happens when the effect ACK arrives and there are no remaining unresolved gate hooks at the same order.

In practice this means: if you see an effect hook showing ACKed while a gate hook at the same order is still Dispatched, that is correct behaviour. The order has not advanced. The gate hook owns advancement for that order — when it ACKs, the engine checks both hooks are resolved and then moves to the next order.

Do not mix gate + effect at the same order if the gate carries a complete code The validator warns about this (AMBIGUOUS_ORDER). If the gate succeeds and skips remaining gates, the effect at the same order has already been dispatched and will still run — but the ordering semantics become harder to reason about. Keep gate-with-code hooks at a dedicated order value.

Policy JSON — Complete Codes

Complete codes appear in two places: on rules (transition-level) and on individual gate hooks. They tell the interpreter which event to fire automatically after a success or failure outcome. 

// On a rule (transition-level) — governs the transition handler outcome
"complete": { "success": 2007, "failure": 2008 }

// On a hook — governs this specific gate hook only
"complete": { "success": 2001, "failure": 2002 }

Transition handler — success waits, failure fires immediately

The transition handler always runs first. But a successful result does not fire the success complete code immediately — the interpreter must run all hooks first. Only after every hook has completed is the transition's success code considered, and even then only if no gate hook has already captured a success code of its own.

Failure is the opposite. If the transition handler fails, the failure complete code is fired immediately and all hooks are skipped. There is no purpose in running hooks if the transition itself did not succeed.

Transition handler outcomeInterpreter behaviour
Returns success Hold the success complete code. Run all hooks in order. Fire the success code only if no hook terminates the chain first — it is the final fallback after all hooks pass through silently.
Returns failure Fire the failure complete code immediately. Skip all hooks. If no failure code is defined, return blocked (HFR3004).

Gate hook success drains effects before firing

When a gate hook succeeds with a success code, the interpreter does not fire that code immediately. Instead it: skips all remaining undispatched gate hooks, runs all remaining effect hooks in order, and only then fires the gate's success code. Effect hooks are side-effects that must still execute regardless of the gate decision.

Gate hook failure is different — failure fires the failure code immediately and skips everything (both remaining gate and effect hooks).

The transition handler's success code is only a fallback. It fires after all hooks complete with no gate termination. If a gate hook fires a success code, the transition handler's success code is discarded.

Gate failure terminates immediately; gate success drains effects first A gate failure complete code fires immediately and skips all remaining hooks. A gate success complete code runs all remaining effect hooks first, then fires. The only pass-through case is a gate hook with no complete codes: on success it continues to the next hook, on failure it rolls back state.
Complete codes do not cascade Rule-level complete codes apply only to the transition handler. They are not inherited by hooks. Each hook's complete codes come exclusively from its own complete block. If a hook has no complete block, it has no auto-advance codes — period.

Policy JSON — Parameter Inheritance

Parameters flow from the catalog to handlers via a two-level inheritance:

ContextParams used
Transition handlerRule-level params
Hook handler — hook has own paramsHook's own params
Hook handler — hook has no paramsInherits parent rule's params

This means a rule can define "params": ["PARAMS.LOAN.MANAGER.APPROVAL"] and all hooks that don't specify their own params will automatically receive the manager approval parameter — no repetition required.

Policy JSON — Timeouts

The optional timeouts array defines automatic event firing when an entity remains in a state beyond a specified duration. Two forms of duration are supported — use whichever is more readable: 

"timeouts": [
  // ISO 8601 duration — fires once (default mode)
  { "state": "RevisionRequired", "timeout": "P14D", "timeout_event": 3012 },

  // ISO 8601 duration — repeat mode (fires again each period until state changes)
  { "state": "ManagerApproval", "timeout": "P1D", "timeout_event": 2009, "timeout_mode": "repeat" },

  // Minutes shorthand — repeat with a cap on retries
  { "state": "ReworkRequired", "timeout_minutes": 2, "timeout_event": 4010, "timeout_mode": "repeat", "max_retry": 4 }
]
FieldTypeRequiredDescription
statestringrequired The state the entity must be dwelling in for this timeout to apply.
timeoutstringone of the two Duration as an ISO 8601 period string. e.g. "P1D" (1 day), "P2D" (2 days), "PT30M" (30 minutes), "P14D" (14 days). Mutually exclusive with timeout_minutes.
timeout_minutesintegerone of the two Duration in minutes as a plain integer. Convenience alternative to timeout for short durations. e.g. 2 means 2 minutes. Mutually exclusive with timeout.
timeout_eventintegerrequired The event code fired when the timeout elapses. Must be a valid event code in the definition.
timeout_modestringoptional Default: "once". "once" — fires the event exactly once, then stops. "repeat" — fires the event repeatedly, once per period, until the entity leaves the state or max_retry is reached.
max_retryintegeroptional Maximum number of times the timeout event may fire. Only meaningful with "repeat" mode. If omitted, repeat mode fires indefinitely until state changes.
Timeouts and via rules work together A timeout fires an event code — that event causes a transition, and the arriving state may have a via-specific rule targeting exactly that event. This is the standard pattern for reminder/escalation cycles: the timeout fires a self-loop event, the via rule on that event triggers reminder hooks, and repeat mode keeps firing until the entity moves on.

Execution Contract — Order

When an event fires, a compliant interpreter must execute the following steps in order:

  1. Resolve the current state of the entity (or the initial state if first event).
  2. Look up the transition: from == currentState AND event == eventCode.
  3. If no transition found: reject the event.
  4. Run the transition handler with the entity context.
  5. Advance currentState to toState.
  6. Find the one matching rule for toState: a via-specific rule takes priority; a general rule is the fallback if no via-specific rule matched.
  7. Execute all hooks from matching rules in order sequence.
  8. After all hooks complete: fire any pending success event.

Execution Contract — Transition Handler

OutcomeAction
Handler returns failure Skip ALL hooks. Fire CompleteFailureCode immediately if set. If not set, the execution stops — no auto-advance.
Handler returns success Hold CompleteSuccessCode. Run all hooks first. Fire success code only after all hooks complete.

Execution Contract — Hook Execution

OutcomeHas success codeHas failure codeAction
Effect hook — any result—— Continue to next hook. Result and complete codes are ignored entirely.
Gate hook succeedsyes— Skip remaining gate hooks. Run all remaining effect hooks in order. Fire success code after effects complete.
Gate hook succeedsno— Continue to next hook. No termination.
Gate hook fails—yes Terminate immediately. Skip ALL remaining hooks (gate and effect). Fire failure code now.
Gate hook fails—no Roll back currentState to fromState. Return blocked (HFR3005). No auto-advance.
Gate failure terminates immediately; gate success drains effects A gate failure complete code fires immediately and skips all remaining hooks. A gate success complete code skips remaining gates but still runs all effect hooks — only terminating after the effects complete. Only a gate hook with no complete codes passes through silently.

Why success also terminates — the escalation pattern

Consider a multi-stage approval where you want to try reviewers in sequence and stop as soon as one approves. If success codes deferred execution (running all remaining hooks first), you could never short-circuit the escalation chain. With immediate termination on success, the pattern works naturally: 

// Multi-stage approval — stop at first approver, escalate on failure
"emit": [
  {
    "route":    "APP.APPROVAL.STAGE1",
    "type": "gate",
    "order":    1,
    "complete": { "success": 5001 }   // ← success terminates; stage 2 never runs if stage 1 approves
  },
  {
    "route":    "APP.APPROVAL.STAGE2",
    "type": "gate",
    "order":    2,
    "complete": { "success": 5001 }   // ← only reached if stage 1 had no success code or did not terminate
  },
  {
    "route":    "APP.APPROVAL.STAGE3",
    "type": "gate",
    "order":    3,
    "complete": { "success": 5001, "failure": 5002 }
  }
]

Stage 1 runs. If it succeeds and has a success code → terminates immediately, fires event 5001, stages 2 and 3 never run.
If stage 1 has no success code (or succeeds without one) → stage 2 runs.
Stage 3 has both codes → it is a full terminator; anything after it would be unreachable.

A hook with no complete code is a pass-through

A gate hook with no complete block participates in the chain but never terminates it. On success it silently passes to the next hook. On failure it rolls back and returns blocked — the caller's Tier 2 or Tier 3 code must decide what happens next. 

// Archive hook — must succeed but does not drive auto-advance
{
  "route":    "APP.LOAN.MANAGER.DOC.ARCHIVE",
  "type": "gate",
  "order":    2
  // no complete block — success passes through, failure rolls back + blocked
}
Transition success code is the final fallback The transition handler's own success code only fires if no hook terminated the chain first. If any gate hook fires a success code, the transition's success code is discarded. If no hook terminates, the transition's code is the last resort after all hooks pass through.

Execution Contract — Parameter Resolution

Before calling any handler, the interpreter resolves the parameters available to it:

Handler typeParameters provided
Transition handlerFull SnapshotParameter objects for each code in the rule's params list.
Hook handler (hook has own params)Full objects for the hook's own param codes.
Hook handler (hook has no params)Inherits — full objects for the parent rule's param codes.
Hook handler (neither hook nor rule has params)Empty list.

Execution Contract — Auto-Advance

Auto-advance means the interpreter automatically fires the next event without the caller having to provide it. This is entirely driven by complete codes in the policy JSON.

Auto-advance is recursive — if the fired event itself results in a complete code, the interpreter continues advancing until no more complete codes are resolved or a terminal state is reached.

Cycle risk If complete codes form a cycle (A → B → A → ...) with no terminal condition, the interpreter will loop indefinitely. Authors must ensure complete codes always lead toward terminal states or states with no further complete codes.

Validation — Structural Rules

A compliant interpreter must validate the definition JSON before accepting it for execution. Errors must cause registration to be rejected. Warnings are informational and may be surfaced to the author without rejection.

CodeSeverityCondition
NO_INITIAL_STATEErrorNo state is marked is_initial.
MULTIPLE_INITIAL_STATESErrorMore than one state is marked is_initial.
UNKNOWN_FROM_STATEErrorA transition's from references a state not in the states list.
UNKNOWN_TO_STATEErrorA transition's to references a state not in the states list.
DUPLICATE_EVENT_CODEErrorThe same event code appears more than once from the same state.
TERMINAL_HAS_TRANSITIONSErrorA state marked is_final has outgoing transitions.
STUCK_STATEErrorA non-terminal state has no outgoing transitions — entities will be permanently stuck.
CIRCULAR_ONLYErrorA state's only outgoing transition is a self-loop — guaranteed infinite loop with no exit.
UNREACHABLE_STATEWarningA state has no incoming transitions and is not the initial state — it can never be reached.
SELF_LOOP_WITH_EXITWarningA state has a self-loop but also has exit transitions — likely intentional, verify it is not accidental.

Validation — Policy Rules

CodeSeverityCondition
POLICY_UNKNOWN_STATEWarningA policy rule targets a state that does not exist in the definition — the rule will never fire.
AMBIGUOUS_ORDERErrorMultiple gate hooks at the same order value define complete codes — execution order is undefined.
EFFECT_WITH_COMPLETEWarningAn effect hook defines complete codes — they will be ignored at runtime.
UNREACHABLE_HOOKWarningA gate hook appears after another gate hook that has both success and failure complete codes — it can never execute. Effect hooks after such a gate terminator are still reachable (gate success drains effects first).
NO_FAILURE_PATHWarningA gate hook has no failure complete code and the transition has no failure complete code — on failure, the interpreter cannot auto-advance. The application must handle it explicitly.

Error Codes

Every error, warning, and blocked condition in HaleyFlow is identified by a structured error code of the form HFXnnnn where X is a letter denoting the subsystem and nnnn is a four-digit number. Codes ending in 1xx within a range are warnings; all others are errors unless explicitly noted.

RangeSubsystemWhen raised
HFD1xxxDefinition StructuralAt definition registration / relay initialization — structural checks on states and transitions
HFP2xxxPolicy Hook / RuleAt definition registration / relay initialization — policy hook and rule checks
HFR3xxxRelay RuntimeAt runtime inside NextAsync — blocked conditions during relay execution
HFB4xxxBackfill ValidationAt backfill import time — validation of backfill objects against the definition
HFF5xxxFlowBus / Relay HostAt initiation time — routing and registration errors in FlowBus and WorkflowRelayHost

HFD1xxx — Definition Structural

Raised by the policy validator when parsing a definition. Errors in this range reject registration entirely.

CodeNameSeverityDescription
HFD1001NO_INITIAL_STATE Error No state is marked is_initial: true. The interpreter cannot determine the entry point.
HFD1002MULTIPLE_INITIAL_STATES Error More than one state is marked initial. Only exactly one is permitted.
HFD1003UNKNOWN_FROM_STATE Error A transition references a from state that does not exist in the states list. Likely a spelling mistake.
HFD1004UNKNOWN_TO_STATE Error A transition references a to state that does not exist in the states list. Likely a spelling mistake.
HFD1005DUPLICATE_EVENT_CODE Error The same event code appears more than once from the same state. Routing is ambiguous — the interpreter cannot determine which transition to follow.
HFD1006TERMINAL_HAS_TRANSITIONS Error A state marked is_final: true has outgoing transitions. A terminal state cannot be left.
HFD1007STUCK_STATE Error A non-terminal state has no outgoing transitions. Any entity that reaches this state will be permanently stuck.
HFD1008CIRCULAR_ONLY Error A state's only outgoing transition is a self-loop back to itself. An infinite loop is guaranteed with no exit path.
HFD1101UNREACHABLE_STATE Warning A state has no incoming transitions and is not the initial state. It can never be reached during execution.
HFD1102SELF_LOOP_WITH_EXIT Warning A state has a self-loop transition but also has exit transitions. Likely intentional (e.g. reminder ping, timeout retry) — verify it is not accidental.

HFP2xxx — Policy Hook / Rule

Raised by the policy validator when checking hook configuration. Errors reject registration; warnings are informational.

CodeNameSeverityDescription
HFP2001POLICY_UNKNOWN_STATE Warning A policy rule targets a state that does not exist in the definition. The rule is a dead letter — it will never fire.
HFP2002AMBIGUOUS_ORDER Error Multiple gate hooks at the same order value define complete codes. Execution order within the same order value is undefined — the interpreter cannot guarantee which fires first.
HFP2101EFFECT_WITH_COMPLETE Warning An effect hook defines complete codes. Effect hooks cannot drive auto-advance — these codes are silently ignored at runtime.
HFP2102UNREACHABLE_HOOK Warning A gate hook appears after a "full terminator" — a prior gate hook that defines both success and failure complete codes. Since all paths from that gate terminator lead away, this gate hook can never be reached. Effect hooks after a gate terminator are still reachable because gate success drains effects before firing its code.
HFP2103NO_FAILURE_PATH Warning A gate hook has no failure complete code, and the transition also has no failure complete code. If this hook fails, the interpreter cannot auto-advance — a code-side Tier 2 or Tier 3 handler must decide the next step.

HFR3xxx — Relay Runtime

Raised at runtime inside NextAsync. These are blocked conditions — execution stopped and the current state was not advanced (or was rolled back).

CodeConditionDescription
HFR3001No initial state at runtime NextAsync was called with an empty CurrentState but the definition has no state marked initial. Cannot resolve the starting point.
HFR3002Invalid transition No transition exists from the current state via the supplied event code. The event is not valid from where the entity currently is.
HFR3003Blocked by monitor The monitor callback returned false. Execution was blocked before the handler ran. No state change occurred.
HFR3004Transition handler failed The transition handler returned false and no failure complete code was available to auto-advance. All hooks were skipped.
HFR3005Blocking hook failed A gate hook returned false and no failure path (hook-own or transition-level) was available. CurrentState was rolled back to the from state.

HFB4xxx — Backfill Validation

Raised when validating a WorkflowBackfillObject before importing historical instance data.

CodeSeverityDescription
HFB4001Error WorkflowName is missing on the backfill object.
HFB4002Error EntityRef is missing on the backfill object.
HFB4003Error The backfill object contains no transitions. At least one is required.
HFB4004Error The workflow definition was not found in the engine for the given name and environment code.
HFB4005Error A transition in the backfill object (FromState → ToState via EventCode) is not a valid transition in the definition.
HFB4101Warning A hook entry in the backfill object has an empty Route — it will be skipped during import.
HFB4102Warning A hook route in the backfill object is not defined in the policy for this transition. It will be recorded as-is (legacy data tolerance).

HFF5xxx — FlowBus / Relay Host

Raised at initiation time by the FlowBus router or the relay host when routing or registration problems are detected.

CodeDescription
HFF5001 No workflow executor is registered. Neither an engine consumer service nor a relay service is present in DI.
HFF5002 Engine mode was explicitly requested but the engine consumer service is not registered in DI.
HFF5003 Engine mode requires a StartEvent but none was provided in the initiation request.
HFF5004 The engine rejected the initiation request. See the engine's own reason for details.
HFF5005 Relay mode was explicitly requested but the relay service is not registered in DI.
HFF5006 No relay is registered for the requested workflow name. The relay host could not find a matching WorkflowRelayBase subclass.
HFF5007 The StartEvent value is not a valid integer event code. Relay mode requires a numeric event code, not an event name string.
