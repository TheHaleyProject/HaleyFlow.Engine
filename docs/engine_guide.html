Haley Workflow Engine

A reliable, DB-backed workflow engine for .NET. Define workflows as JSON policies. Fire events to advance state. React to hook emissions in your services. The engine handles persistence, delivery guarantees, retry, and timeout — your code handles business logic.

Getting Started

Quick Start

// 1. Build engine — auto-creates/migrates DB schema on first run
var engine = await new WorkFlowEngineMaker {
    Options = new WorkFlowEngineOptions {
        AckGateEnabled           = true,
        DefaultStateStaleDuration = TimeSpan.FromHours(24),
        MaxRetryCount             = 5
    }
}.WithConnectionString(connectionString).Build(adapterGateway);

// 2. Subscribe to events and notices
engine.EventRaised  += HandleEventAsync;
engine.NoticeRaised += HandleNoticeAsync;

// 3. Import schema and policy (idempotent — safe every startup)
await engine.ImportDefinitionJsonAsync(envCode, "Production", defJson);
await engine.ImportPolicyJsonAsync(envCode, "Production", policyJson);

// 4. Register consumer + start heartbeat
var consumerId = await engine.RegisterConsumerAsync(envCode, myGuid);
_ = Task.Run(() => BeatLoop(engine, envCode, myGuid, cts.Token)); // every ~10s

// 5. Start monitor
await engine.StartMonitorAsync(cts.Token);

// 6. Trigger a transition
var result = await engine.TriggerAsync(new LifeCycleTriggerRequest {
    EnvCode     = envCode,
    DefName     = "loan-approval",
    EntityId    = loanId.ToString(),
    Event       = "submit",
    Actor       = userId,
    AckRequired = true,
    OccurredAt  = null,        // set DateTimeOffset for replay
    SkipAckGate = false        // set true to bypass ACK gate once
});

if (!result.Applied) {
    Console.WriteLine($"Blocked: {result.Reason}"); // e.g. "BlockedByPendingAck"
    return;
}

// 7. In event handler — ACK after processing
async Task HandleEventAsync(ILifeCycleEvent e) {
    if (e is ILifeCycleHookEvent hook) {
        // dispatch based on hook.Route
        await DispatchHookAsync(hook);
        await engine.AckAsync(envCode, myGuid, hook.AckGuid, AckOutcome.Processed);
        // ↑ if hook belongs to a group and all siblings are Processed,
        //   HOOK_GROUP_COMPLETE notice fires automatically
    }
}

// 8. Handle notices
async Task HandleNoticeAsync(LifeCycleNotice n) {
    switch (n.Code) {
        case "HOOK_GROUP_COMPLETE":
            var group    = (string)n.Data["groupName"];
            var instGuid = (string)n.Data["instanceGuid"];
            // all hooks in group done — trigger next macro event
            await engine.TriggerAsync(new LifeCycleTriggerRequest {
                EntityId = instGuid, Event = "review_complete", /* ... */
            });
            break;
        case "ACK_SUSPEND":
            logger.LogCritical("Instance suspended: {msg}", n.Message);
            break;
        case "STATE_STALE":
            logger.LogWarning("Instance overdue: {guid}", n.Data["instanceGuid"]);
            break;
    }
}C#
Concept

Concept & Objective

The engine is intentionally "dumb" — it enforces state transitions and tracks delivery, but carries no business logic.

✅ What the engine does
  • Transitions instances through states via triggers (macro workflow)
  • Emits lifecycle transition events (consumers decide what to do)
  • Emits hook events — directly tells consumers which method to call
  • Tracks delivery/processing via ACKs and re-sends until completed
  • Produces timeline/reporting
  • Auto-triggers timeout events when configured
  • Fires advisory notices for stale states without timeouts
🚫 Non-goals
  • Business logic — lives in the application + JSON policy/routes
  • Micro-workflow (use hooks for that)
  • Auto-advancing state without a consumer trigger
  • Runtime state ownership — no "instance owner" concept
Lifecycle vs. Hook
Lifecycle events tell consumers "the state changed" — the consumer decides what to do. Hook events directly instruct "call this route" — configured via policy JSON, no hard-coding in the application.
Why Workflow?

Why a Workflow Engine?

Every significant business entity has a lifecycle — a user isn't just a row in a database, they're a thing that was created, onboarded, activated, maybe churned, maybe reactivated. The moment your process spans multiple steps, multiple systems, and any amount of time, you need something more than an API call and a background job.

The Common Problem
You try to cover a few steps in the API call and defer the rest to background jobs. This works until your system crashes mid-process. Now you don't know what ran and what didn't. Your background jobs eventually catch up, but the window of uncertainty causes side effects, duplicate work, and support tickets. A workflow engine eliminates that window.

The Core Pattern

A workflow engine is the right tool whenever a process has all of these characteristics:

  • Multiple steps that must happen in sequence or in parallel
  • Steps span time — from seconds to months
  • Steps involve external systems that can fail independently
  • Partial completion is a real problem — you can't roll back a sent email or a created account
  • Business visibility is required — someone needs to ask "where is this entity right now?"

Use Cases by Domain

👤 Onboarding & Registration

User signs up → create account → provision resources → send welcome email → wait for first login → if no login in 3 days → send nudge → assign account manager → escalate. Every step is tracked. Crash mid-way? Resume from where it stopped. Business can see exactly which step each user is in.

✅ Approval Workflows

Expense reports, document reviews, legal sign-offs. If a manager hasn't approved in 48 hours, auto-escalate. No "did this get approved?" mystery. Full audit trail for compliance.

📦 Order Fulfillment & E-Commerce

Order placed → payment captured → inventory reserved → warehouse picks → shipped → delivered → return window tracked. If payment fails, the order does not proceed to fulfillment. If shipping fails, retry or notify. You know exactly which orders are stuck at which step.

🏦 Loan / Insurance / Credit Applications

Application submitted → document collection → verification → underwriting → decision → offer sent → customer acceptance → disbursement. Steps can take days or weeks across multiple external systems. Enforce "don't proceed to underwriting until all documents are verified." Timeout rules: "if customer hasn't accepted the offer in 7 days, expire it."

🛡️ KYC & Compliance

Identity submitted → ID verification (3rd party) → AML/sanctions check → risk scoring → manual review if flagged → approved or rejected. Mandatory steps cannot be skipped. If a third-party API is down, the process parks and retries. The compliance audit trail is automatic.

🔄 SaaS Subscription Lifecycle

Trial starts → usage tracked → trial-expiring notice (D-3) → trial expires → convert or cancel → if converted: billing setup → if cancelled: offboarding → data retention window → purge. Time-based transitions are automatic. No cron jobs to pray over. State is always known.

⚙️ DevOps & Infrastructure Provisioning

Tenant signs up → provision database → provision storage → configure DNS → run smoke tests → enable access → send credentials. If any step fails you have a half-provisioned tenant. A workflow engine gives you compensating actions on failure and a clean retry or rollback path.

🎫 Support Ticket & Incident Management

Ticket created → auto-categorised → assigned to tier-1 → if unresolved in 2 hrs → escalate to tier-2 → if unresolved in 4 hrs → page on-call → resolved → post-mortem scheduled. SLA timers are first-class. Escalation is automatic, not "someone forgot to check."

🏥 Healthcare & Patient Journeys

Patient referred → appointment scheduled → pre-appointment forms → appointment → follow-up → prescription issued → pharmacy notified → refill reminder. Auditable trail, mandatory step enforcement, and timeouts for follow-ups.

📰 Content Publishing Pipelines

Content created → editorial review → legal review → SEO check → scheduled for publish → published → performance tracked → archive after N months. Nothing goes live without passing all gates. Rollback is a defined state, not a manual operation.

The Litmus Test
The moment you find yourself writing "a background job that checks if X happened and does Y if it didn't" — that is a workflow engine use case waiting to happen. The background job is your retry logic; the engine makes that retry logic crash-safe, auditable, and observable without any extra code.

3.2 Engine Evaluates + Persists (inside one transaction)

  1. 1
    Load blueprint (IBlueprintManager) — cached read. Done outside the transaction.
  2. 2
    Resolve consumers — at least one transition consumer must exist. Hook consumers are optional.
  3. 3
    Resolve policy (IPolicyEnforcer.ResolvePolicyAsync) — gets the latest policy for the definition.
  4. 4
    Ensure instance (IStateMachine.EnsureInstanceAsync) — creates or retrieves. Policy is attached at creation time and never changes for that instance.
  5. 4b
    ACK gate check — if AckGateEnabled && !SkipAckGate, counts ack_consumer rows for the last lifecycle entry of this instance that are NOT in terminal status (Processed / Failed). If any exist, commits and returns Applied=false, Reason="BlockedByPendingAck".
  6. 5
    Apply transition (IStateMachine.ApplyTransitionAsync) — validates transition, CAS-updates instance state, inserts lifecycle row. If already moved → commits and returns early.
  7. 6
    Resolve rule context from policy JSON — extracts params, on_success_event, on_failure_event for the matched rule.
  8. 7
    Create lifecycle ACK (IAckManager.CreateLifecycleAckAsync) — one ack_guid, multiple consumer rows, linked via lc_ack.
  9. 8
    Emit hooks (IPolicyEnforcer.EmitHooksAsync) — upserts hook rows. Resolves route string → hook_route.id (FK). Resolves group name → hook_group.id (FK, nullable). Computes minOrder across all matched emit entries. Hooks at order > minOrder are written with dispatched=0 (invisible to the monitor; waiting for prior-order blocking hooks to complete). Only min-order hooks are returned in the LifeCycleHookEmission list for step 9. If AckRequired=false, all hooks dispatch immediately regardless of order.
  10. 9
    Create hook ACKs (IAckManager.CreateHookAckAsync) — one ack_guid per dispatched hook, multiple consumer rows, linked via hook_ack. Higher-order hooks (dispatched=0) get no ACK rows here; rows are created later by AdvanceNextHookOrderAsync when prior blocking hooks complete.
  11. 10
    Commit first. Only after commit does the engine raise C# events via FireEvent. Dispatch failures become notices; monitor will re-send via pending ACK rows.

3.2b ACK Gate

Prevents a new state transition from being applied while the last lifecycle entry of the instance still has unresolved ACK consumers.

Opt-in (disabled by default)
new WorkFlowEngineOptions {
    AckGateEnabled = true
};
Per-request bypass
req.SkipAckGate = true;
// force transition regardless of pending ACKs

When blocked, TriggerAsync returns:

{ Applied: false, Reason: "BlockedByPendingAck", InstanceGuid: "..." }
Use case
Enable when strict ordering is required — a downstream system must fully process a transition before the workflow can advance. Leave disabled for high-throughput flows where ordering is application-managed.

How Retry and trigger_count interact

trigger_count tracks how many times the engine has actually re-delivered the event. It is incremented only by the monitor when it fires the event — not by the consumer calling Retry.

Actiontrigger_countnext_due
Engine first fires event+1now + pendingWindow
Consumer calls AckAsync(Retry, retryAt)no changeretryAt (consumer-controlled)
Monitor ticks — next_due still in futureno changeunchanged
Monitor ticks — next_due has passed → re-delivers+1now + pendingWindow
Consumer calls Retry againno changenew retryAt
Monitor re-delivers again+1now + pendingWindow
Retry does not grant unlimited time

The consumer can push the re-delivery date forward using retryAt, but each time the monitor eventually re-delivers, trigger_count increments. Once trigger_count ≥ MaxRetryCount, the engine marks the ACK Failed and suspends the instance (for blocking hooks) regardless of how far out the consumer kept scheduling retries.

For genuinely long-running human tasks (reviews that take days or weeks), the right pattern is to ACK Processed immediately when the hook fires — recording that the task was handed off — and then call TriggerAsync with the appropriate event when the human completes their work. This keeps the ACK clean and the workflow from being held hostage to a retry counter.

Hook Group Completion
After any AckAsync(Processed) call, the engine automatically checks whether the ACK belongs to a grouped hook. If all sibling hooks in the group (same instance_id + state_id + via_event + on_entry) are now Processed, a HOOK_GROUP_COMPLETE notice fires. Errors in this check are isolated — the ACK itself succeeds regardless.

3.5 Reliability: Monitor

LifeCycleMonitor runs on a PeriodicTimer. Single active execution guaranteed via Interlocked gate. Each tick runs three jobs:

A) Default Staleness "OverDue" — runs first

An instance is returned by the stale scan only when all five conditions hold simultaneously:

  1. Instance status is not Suspended, Completed, Failed, or Archived
  2. No policy timeout is configured for the current state — if a timeout exists, the timeout job owns that state; the stale scan skips it entirely
  3. The instance has been in its current state longer than WorkFlowEngineOptions.DefaultStateStaleDuration
  4. No lifecycle ACK consumer row exists with status ≠ Processed (i.e. all lifecycle ACKs are terminal)
  5. No hook ACK consumer row exists for this instance+state with status ≠ Processed (i.e. all hook ACKs are terminal)
What conditions 4 & 5 mean in practice
If there is any open ACK (lifecycle or hook, Pending or Delivered) the stale scan silently skips the instance. The ACK re-dispatch job (section B below) is already handling it. STATE_STALE fires only when the instance is genuinely stuck with nothing outstanding.

No DB writes occur during the stale scan. An in-memory throttle prevents the same notice from firing on every monitor tick.

B) ACK Re-dispatch — per consumer

Step 1: PushNextDueForDownAsync — pushes next_due forward for consumers with stale heartbeat, without incrementing trigger_count. Dead services don't burn max retries.

Step 2: For each due ACK (next_due <= now):

  • If TriggerCount >= MaxRetryCount:
    • Mark ACK Failed (next_due = NULL)
    • If hook was blocking: suspend instance + fire ACK_SUSPEND
    • If hook was non-blocking: fire ACK_FAIL notice, instance continues
  • Otherwise: increment trigger count, set next next_due, fire ACK_RETRY, re-raise event

C) Policy-Defined Timeout Processing

States with a timeouts policy rule are owned exclusively by this job — the stale scan (job A) skips them entirely. The lc_timeout table acts as an idempotency marker. Resolution in both cases is natural: once the instance transitions out, the query stops returning it (no explicit "mark resolved" step needed).

Case A — timeout_event is set

The engine auto-moves the instance without human intervention:

  1. INSERT into lc_timeout first (idempotent) — marker is written before TriggerAsync so a mid-process crash cannot cause a double-fire on the next tick
  2. TriggerAsync(entityId, event = timeout_event_code, actor = "engine.monitor") — goes through the full transactional pipeline: new lifecycle entry, hooks, ACKs, dispatch
  3. Fire TIMEOUT_FIRED notice (Info) for observability
Case B — no timeout_event (advisory escalation)

The engine raises STATE_TIMEOUT_EXCEEDED notices on a schedule but does not auto-move the instance. Someone must act on the notice to advance the workflow.

The lc_timeout table tracks per-entry scheduling state. Field names mirror ack_consumer for consistency:

  • trigger_count — how many notices have been sent so far
  • last_trigger — when the last notice was fired (audit)
  • next_due — when the next notice should fire
  1. First occurrence: INSERT lc_timeout(lc_id, trigger_count=1, last_trigger=now, next_due=now+DefaultStateStaleDuration) + fire STATE_TIMEOUT_EXCEEDED
  2. Subsequent ticks: if next_due <= UTC_NOW → increment trigger_count, update last_trigger and next_due, fire notice again with triggerCount in Data
  3. If max_retry is defined on the timeout rule (or falls back to engine-level MaxRetryCount) and trigger_count >= max_retry: call FailWithMessageAsync(instance) + fire STATE_TIMEOUT_FAILED
Repeat interval uses DefaultStateStaleDuration, not the policy duration
The policy timeout duration is the initial grace period — how long to wait before the first notice. After that, DefaultStateStaleDuration (engine config) controls the repeat cadence (e.g. every 12 h). This way a 7-day grace period doesn't mean waiting another 7 days between each escalation notice.
The timeout_mode field
  • once (default) — Case A: fire event once. Case B: send one notice then stop (trigger_count capped at 1).
  • repeat — Case A: not meaningful (instance moves state anyway). Case B: keep notifying using next_due scheduling at DefaultStateStaleDuration intervals.
Implementation status — job C is pending
The infrastructure exists: QRY_LC_TIMEOUT.LIST_DUE_PAGED, MariaLifeCycleTimeoutDAL, and the lc_timeout table (with trigger_count, last_trigger, next_due) are all in place. What is missing: IWorkFlowDAL.LcTimeout property and the wiring in MonitorOrchestrator.RunMonitorOnceInternalAsync. The notice codes TIMEOUT_FIRED, STATE_TIMEOUT_EXCEEDED, and STATE_TIMEOUT_FAILED are reserved but not yet emitted.
Architecture

Architecture

For SDK users
This section describes the engine's internal architecture. If you are building an application that uses the engine, you do not need to understand these components — they are wired up automatically by WorkFlowEngine. Jump to Consumer Patterns, Policy Reference, or Examples.

4.1 WorkFlowDAL DB gateway

  • Single entry point to DB for all engine operations
  • Consolidates all read/write DALs behind IWorkFlowDAL
  • Enforces query-first discipline (QRY_*)
  • Exposes CreateNewTransaction() for engine-owned transactions
  • DAL properties: Instance, LifeCycle, LcAck, HookAck, AckConsumer, Hook, HookRoute , HookGroup , Blueprint, Policy, …

4.2 BlueprintImporter

Role: Import-time canonicalization (JSON → SQL).

  • Loads definition JSON: states / events / transitions / categories / timeouts
  • Loads policy JSON: rules / hooks / emit entries / params / timeouts
  • Writes normalized SQL rows in one transaction
  • Policy hashed (SHA-256) for deduplication — same content = same policy row
  • Timeouts parsed from ISO 8601 ("P2D") or direct timeout_minutes

4.3 BlueprintManager

Role: Runtime blueprint read + cache.

  • Builds LifeCycleBlueprint: StatesById, EventsById, EventsByCode, Transitions map, InitialStateId
  • Caches by (env_code, def_name) and by def_version_id
  • Supports invalidation after imports: InvalidateAsync()
  • Important: do not poison cache with a canceled task (avoid capturing ct inside cached lazy)
  • Consumer management: EnsureConsumerIdAsync, BeatConsumerAsync

4.4 StateMachine

Role: Pure state transition executor (dumb by design).

  • Ensures instance exists for (def_version_id, entity_id)
  • Resolves event by code or name
  • Validates transition exists in the definition
  • Updates instance.current_state using CAS to avoid race conflicts
  • Writes lifecycle + lc_data rows (includes occurred column for replay)
  • Returns ApplyTransitionResult (Applied / Reason / From / To / Event / LifeCycleId)

4.5 PolicyEnforcer

Role: Policy evaluation + hook emission.

  • ResolvePolicyAsync(definitionId) — latest policy for a definition
  • ResolvePolicyByIdAsync(policyId) — specific policy (for instance-attached policies)
  • ResolveRuleContextFromJson(policyJson, toState, viaEvent) — returns RuleContext with params, success/failure events
  • ResolveHookContextFromJson(policyJson, toState, viaEvent, hookCode) — hook-specific context including not_before, deadline
  • EmitHooksAsync(blueprint, instance, transition, load, policyJson):
    • Matches policy rules by (state, via_event)
    • For each emit entry: resolves route string → hook_route.id (upsert, global table)
    • If "group" present: resolves group name → hook_group.id (upsert, global table)
    • Upserts hook row with route_id, blocking, group_id
    • Returns LifeCycleHookEmission with: HookId, HookCode, OnEntry, IsBlocking , GroupName , OnSuccessEvent, OnFailureEvent, NotBefore, Deadline, Params
Blocking inheritance chain
emit.blocking → rule.blocking → default true. The emit-level value wins if present.

4.6 AckManager

Role: ACK persistence + consumer tracking + monitor-ready dispatch lists.

  • Creates ack records and links: lc_ack (lifecycle_id → ack_id), hook_ack (hook_id → ack_id)
  • Creates ack_consumer rows with initial status and next_due scheduling
  • Insert-only rule: EnsureConsumersInsertOnlyAsync — never overwrites existing status/next_due. Prevents rescheduling on restart
  • Updates statuses on AckAsync calls
  • Monitor dispatch: ListDueLifecycleDispatchAsync, ListDueHookDispatchAsync (carry group_name )
  • Critical: ACK identifiers are stable — re-dispatch always uses the same ack_guid

4.7 LifeCycleMonitor

Role: Timer + safety gate. Runs WorkFlowEngine.RunMonitorOnceInternalAsync each tick. No business logic lives here — logic is in WorkFlowEngine.

  • Configurable PeriodicTimer interval
  • Single active execution via Interlocked gate on _runGate
  • Implements IAsyncDisposable
  • Supports manual RunOnceAsync(ct) for testing

4.8 WorkFlowEngine

Role: The orchestrator + public API surface. Implements IWorkFlowEngine + ILifeCycleEngine.

Public operations:

  • TriggerAsync — full transactional trigger pipeline (includes ACK gate check)
  • AckAsync — two overloads; after Processed: runs hook group completion check
  • RegisterConsumerAsync, BeatConsumerAsync
  • InvalidateAsync, ClearCacheAsync
  • StartMonitorAsync, StopMonitorAsync
  • GetTimelineJsonAsync
  • UpsertRuntimeAsync — single write API for runtime activity logging (call twice: once with "running", once with final status)

Raises only two C# events: EventRaised (actionable) and NoticeRaised (informational). Implements IAsyncDisposable.

4.9 RuntimeEngine

Role: Observability + app activity/status storage. Not smart workflow.

  • UpsertAsync(RuntimeLogByIdRequest) — internal upsert via ON DUPLICATE KEY UPDATE; called twice (once with "running", once with final status) to update the same row in-place
  • EnsureActivityAsync(name) / EnsureActivityStatusAsync(name) — get-or-create activity and status lookup rows
  • Runtime rows link to: instance_id, state_id, activity_id, status_id, actor_id
  • State is resolved by the public API (WorkFlowEngine.UpsertRuntimeAsync) via AckGuid → hook_ack → hook.state_id, never by the consumer directly
  • Does not participate in workflow correctness — reporting and timeline only

4.10 Timeline & occurred

GetTimelineJsonAsync(LifeCycleInstanceKey key) returns a comprehensive JSON document built via a single SQL query using JSON aggregation. Pass InstanceGuid for direct lookup or EnvCode+DefName+EntityId for name-based resolution.

lifecycle.created
When the engine captured the transition. Always set by the engine. Never overridden.
lifecycle.occurred
The real-world event time. Set by the caller via req.OccurredAt for replay or late-join scenarios. null for normal flow.
Display rule
When rendering a timeline, prefer occurred if present, fall back to created. Never override created.
Consumer Patterns

Consumer Choreography

5.1 Saga Choreography

A decentralized, event-driven process where each consumer (microservice) reacts independently and makes the state transition after completing their work. Other consumers remain as observers.

5.2 No Instance Owner

An instance is not owned by one application. Each consumer owns only the steps it is responsible for. Any consumer may listen; only responsible consumers attempt the next transition.

5.3 "Transition Event" is a Fact, not a Command

A transition event means: "The workflow moved from State A → State B via Event X."

  • Some consumers perform side effects (audit, dashboard, enrichment)
  • Some consumers perform business work and then trigger the next transition (only if within their scope)

5.4 Duplicate Transitions Prevented by the Engine

Multiple consumers may receive the same transition event. The engine prevents duplicate state changes via a Compare-And-Swap (CAS) update on instance.current_state — not via a unique constraint on the lifecycle table.

5.4a The CAS Mechanism

When TriggerAsync applies a transition it executes:

UPDATE instance
   SET   current_state = toState,
         last_event    = eventId
 WHERE id            = instanceId
   AND   current_state = fromState;   -- ← the guardSQL

If two concurrent triggers race, only the first wins (1 row affected). The second gets 0 rows — the engine returns ConcurrencyConflict and inserts no lifecycle row. A retry after success also returns 0 rows (the current_state has already moved), so no duplicate history entry is ever created.

5.4b Why lifecycle Has No Unique Constraint

The lifecycle table is a history log — every row is one distinct state-transition event. A unique constraint on (instance_id, from_state, to_state, event) would be wrong for workflows with loops:

Loops make tuple-level uniqueness impossible
A workflow can legitimately do A → B → A → B. The row (instance, from=A, to=B, event=X) must exist twice — once for each loop iteration. A unique constraint would silently block the second transition and corrupt the workflow.

The CAS on instance.current_state is the idempotency guard. The lifecycle INSERT only runs after the CAS succeeds, so duplicates are structurally impossible without a DB-level unique constraint.

ScenarioWhat happens

Workflow loops back to a previously-visited stateCAS: current_state is the correct from-state again → new row correctly inserted

6.4 Hook Groups

Two or more emit entries sharing the same "group" value in the same rule belong to a hook group. The engine tracks all hooks in the group and fires a notice when all are Processed.

6.5 Ordered Emission

Add an "order" integer (default 1) to each emit entry. The engine groups emits by their order value and dispatches them sequentially:

  1. On trigger, only the lowest-order hooks are dispatched (their hook rows get ACK rows and events fired).
  2. Higher-order hooks are written to the hook table with dispatched=0 — invisible to the monitor until promoted.
  3. When all blocking hooks at the current order reach Processed, the engine atomically creates ACK rows for the next order, marks those hooks dispatched, and fires their events.
  4. If the next order contains only non-blocking hooks, the engine immediately advances through it without waiting for ACKs, continuing until a blocking stage or the end.
AckRequired=false — what it does and when to use it

Effect: When TriggerAsync is called with AckRequired=false:

  • No inbox or outbox ACK rows are created for the consumer to process.
  • Hook ordering is bypassed — all hooks are dispatched immediately at order=1 regardless of their configured order value. Since there is no ACK mechanism to advance through stages, ordered hooks would freeze permanently if ordering were honoured.
  • Hooks that are emitted still create their own hook_ack rows for external consumers — only the transition-level consumer ACK is suppressed.

Legitimate use case — timeout-driven / system-automated transitions:

The engine's timeout monitor fires a TimeoutExpired event automatically. The monitor itself is the processor — there is no application-side consumer business logic to run. You still want outbound notification hooks to fire (APP.NOTIFY.APPLICANT, APP.AUDIT.LOG), but you don't need a consumer to "process" the transition itself. AckRequired=false is correct here.

var result = await engine.TriggerAsync(new LifeCycleTriggerRequest {
    EnvCode    = envCode,
    DefName    = defName,
    EntityId   = entityId,
    Event      = "9000",   // TimeoutExpired
    Actor      = "engine.monitor",
    AckRequired = false    // monitor drives the transition; no consumer pickup needed
}, ct);C#

When NOT to use it:

  • Any trigger that requires consumer-side business logic (approvals, validations, data enrichment). AckRequired=false means the consumer never sees the event.
  • Any workflow where hook ordering matters — the ordering guarantee is silently dropped.
  • Basically all business triggers. The default (AckRequired=true) is almost always correct.

Design insight — AckRequired vs blocking:

AckRequired=false is a coarse instrument: it suppresses everything. If your real need is "this hook should fire but not block the workflow if it fails", use blocking: false on the emit instead. That gives you per-hook fire-and-forget behaviour while keeping the consumer ACK pipeline intact for the hooks that do matter.

6.6 ACK Mode: All vs Any

Controls how many consumers need to ACK Processed for a hook to be considered satisfied.

ack_mode: "all" (default)
  • Every ack_consumer row must reach Processed
  • Each consumer works independently
  • Use when all downstream services must complete (e.g. audit + notify)
ack_mode: "any"
  • First consumer to ACK Processed satisfies the hook
  • All sibling ack_consumer rows are immediately marked Processed
  • Monitor stops retrying the others automatically
  • Use for competing-consumer / any-one-wins patterns (credit bureau fan-out)

Two timeout behaviours

Case A — timeout_event is set: the monitor auto-triggers the event after the duration elapses. The instance moves without human intervention. An lc_timeout marker is written before the trigger for idempotency. The new lifecycle entry shows as a normal transition on the timeline.

Case B — no timeout_event: the monitor raises STATE_TIMEOUT_EXCEEDED notices on a schedule. It does not auto-move the instance. The lc_timeout table tracks trigger_count, last_trigger, and next_due (same field pattern as ack_consumer) so re-noticing survives monitor restarts. The repeat cadence uses DefaultStateStaleDuration — not the policy timeout duration, which is only the initial grace period. Notices stop naturally when the instance transitions out of the state.

timeout_mode values

  • "once" — Case A: fire once (standard). Case B: send one notice then stop.
  • "repeat" — Case A: not meaningful (instance moves state anyway). Case B: keep notifying using DefaultStateStaleDuration as the repeat interval, rescheduling next_due after each notice. The policy timeout duration is the initial grace period only — not the repeat cadence.

Mutual exclusivity with stale scan

The monitor's stale scan (STATE_STALE) explicitly excludes states that have a timeout rule. If you configure a timeout, the stale scan will never fire for that state — the timeout job owns it entirely. This prevents duplicate notices for the same instance.

Several engine subsystems rely on ON DUPLICATE KEY UPDATE (MariaDB/MySQL upsert) to be idempotent. If the matching UNIQUE index is absent, the duplicate-key path never fires and each call creates a new row instead of updating the existing one. The migrations below must be applied once to a fresh or existing schema.

The lifecycle table intentionally has NO unique constraint
lifecycle is an append-only history log. A workflow may loop through the same (from_state, to_state, event) pair multiple times — a unique constraint would silently break loop transitions. Idempotency for lifecycle rows is guaranteed by the CAS on instance.current_state (see §5.4b).

Instance Data Model

Every workflow instance carries three distinct data layers. Understanding which layer to use is critical for correct consumer design.

Why AckGuid — not StateId

State is an engine-internal concept. Consumers only know events (actions like "reviewCompleted") — the engine decides which state that action leads to. Exposing a numeric StateId on the request would force consumers to understand internal DB IDs they have no way to obtain.

Instead, the consumer echoes back the AckGuid it already received in ILifeCycleHookEvent. The engine follows the chain ack.guid → hook_ack.hook_id → hook.state_id to resolve the correct state automatically.

Replay safety: during a replay or a fast-moving workflow the instance may have already moved to a later state by the time the consumer calls UpsertRuntimeAsync. Anchoring via AckGuid ensures the runtime row is always recorded against the state the hook actually fired for, not whatever state the instance happens to be in at call time.

Example 2 — Hook Groups: Parallel Work Gate

Any two emit entries sharing the same "group" value in one rule form a parallel gate. The engine fires HOOK_GROUP_COMPLETE only after all hooks in the group reach Processed.

Design pattern
Put all blocking hooks that must finish together in one group. When the group completes, trigger the next macro event from the HOOK_GROUP_COMPLETE notice handler. Keep advisory/non-critical hooks ungrouped with "blocking": false. 
// Policy: two parallel hooks in "review_checks" group
"emit": [
  { "event": "app.loan.notify_reviewer",  "blocking": true,  "group": "review_checks" },
  { "event": "app.loan.run_credit_check", "blocking": true,  "group": "review_checks" },
  { "event": "app.loan.audit_log",        "blocking": false }  // not in group
]JSON
// Sequence:
// 1. Engine emits notify_reviewer + run_credit_check (both Pending)
// 2. Consumer 1 ACKs notify_reviewer → Processed  (group check: 1/2 done — no notice)
// 3. Consumer 2 ACKs run_credit_check → Processed  (group check: 2/2 done — FIRE!)
// 4. HOOK_GROUP_COMPLETE notice arrives → trigger approve/reject

engine.NoticeRaised += async (n) => {
    if (n.Code != "HOOK_GROUP_COMPLETE") return;
    var group    = (string)n.Data["groupName"];
    var instGuid = (string)n.Data["instanceGuid"];

    if (group == "review_checks") {
        var approved = await _reviewDb.IsApprovedAsync(instGuid);
        await engine.TriggerAsync(new LifeCycleTriggerRequest {
            EntityId = instGuid,
            Event    = approved ? "approve" : "reject",
            EnvCode  = envCode, DefName = "loan-approval", AckRequired = true
        });
    }
};C#

Example 5 — Runtime Activity Tracking

The Consumer Mental Model

A consumer only knows events (actions like "reviewCompleted" or "creditCheckPassed"). It fires an event and the engine decides which state to move to based on the workflow definition. The consumer never needs to know about states, state IDs, or internal DB identifiers — those are all engine-internal concepts.

Runtime logging follows the same principle. The consumer says "I performed this activity with this outcome for this hook" — it does not say "I performed this activity in state 42". The engine derives the correct state automatically.

The Upsert-as-Update Pattern

UpsertRuntimeAsync is the only write API for runtime data. The DB row is uniquely keyed on (instance_id, activity_id, state_id, actor_id) with ON DUPLICATE KEY UPDATE. This means calling it twice with the same key updates the same row — no separate status-update call is needed. 

// Hook fires for "UnderReview" state → consumer handles it:
engine.EventRaised += async (e) => {
    if (e is not ILifeCycleHookEvent h) return;
    if (h.Route != "CREDIT.CHECK") return;

    // Step 1 — record that work has started.
    // AckGuid anchors this entry to the state that fired the hook.
    await engine.UpsertRuntimeAsync(new RuntimeLogByNameRequest {
        Instance = new LifeCycleInstanceKey { InstanceGuid = h.InstanceGuid },
        AckGuid  = h.AckGuid,   // engine resolves state_id from here
        Activity = "credit check",
        ActorId  = h.ConsumerId.ToString(),
        Status   = "running"
    });

    // Step 2 — do the actual work.
    var passed = await _credit.CheckAsync(h.EntityId);

    // Step 3 — update status. Same key → same row updated in-place.
    await engine.UpsertRuntimeAsync(new RuntimeLogByNameRequest {
        Instance = new LifeCycleInstanceKey { InstanceGuid = h.InstanceGuid },
        AckGuid  = h.AckGuid,
        Activity = "credit check",
        ActorId  = h.ConsumerId.ToString(),
        Status   = passed ? "approved" : "rejected",
        Data     = new { Score = _credit.LastScore, PassedAt = DateTimeOffset.UtcNow }
    });

    // Step 4 — ACK the hook so the engine can advance.
    var outcome = passed ? AckOutcome.Processed : AckOutcome.Failed;
    await engine.AckAsync(envCode, myGuid, h.AckGuid, outcome);
};C#
Why AckGuid instead of StateId

The instance may have already moved to a later state by the time the consumer calls UpsertRuntimeAsync (this is common during replay and fast-moving workflows). Passing AckGuid tells the engine "this runtime entry belongs to the hook that sent me this guid" — the engine looks up ack → hook_ack → hook.state_id to get the correct state. Without it, the engine would fall back to instance.current_state, which may already be a later state.

If you are recording activity outside of a hook handler (e.g. from a direct API call where no hook fired), omit AckGuid — the engine will use the current state, which is correct in that case.

Integration

Engine Proxy Architecture

The engine exposes all its capabilities through IWorkFlowEngine, but consumers must never hold a direct reference to it. Instead, every engine interaction goes through ILifeCycleEngineProxy — a single abstraction that isolates the consumer from wherever the engine happens to be running.

Core Rule
The consumer package (Haley.Flow.Consumer) has zero knowledge of IWorkFlowEngine or IWorkFlowEngineAccessor. It only ever holds an ILifeCycleEngineProxy.

Why a Proxy?

The consumer is designed to be deployment-agnostic. The proxy registration is the only thing that changes between deployment topologies — the consumer service itself never changes:

DeploymentProxyWhere proxy livesHow it works
In-process (same host)InProcessEngineProxyEngine packageDirect method calls + in-memory channels for events
Deferred in-processDeferredInProcessEngineProxyEngine packageSame as above but engine resolved lazily on first call
Remote (separate host)HttpEngineProxy (future)Consumer packageHTTP calls to engine's REST API endpoints

Why InProcessEngineProxy lives in the Engine project

InProcessEngineProxy holds a direct reference to IWorkFlowEngine — that direct coupling is precisely why it cannot live in the consumer package. The engine project is the only one allowed to know about engine internals. An HttpEngineProxy (future) makes only HTTP calls and has no engine dependency, so it can live in the consumer package.

ILifeCycleEngineProxy surface

The proxy exposes the complete surface the consumer needs, split across three concerns:

Inherited fromCovers
ILifeCycleConsumerBusRegistration, heartbeat, ACK, blueprint import, GetDefinitionIdAsync
ILifeCycleRuntimeBusTrigger, timeline, instance data & context, runtime log, list/suspend/resume/reopen/fail, pending ACKs
Own membersGetDueTransitionsAsync, GetDueHooksAsync, NoticeRaised event

Engine-management operations (StartMonitorAsync, GetHealthAsync, etc.) are intentionally not on the proxy — they are engine-internal concerns.

Event delivery vs on-demand queries

GetDueTransitionsAsync and GetDueHooksAsync look like queries but they are fundamentally different from every other method on the proxy:

⚡ Event delivery (push → pull)

GetDueTransitions/HooksAsync

Exist because the engine has EventRaised. When the engine fires an event (on trigger, monitor resend, hook advancement), it must reach the consumer somehow.

In-process: events are written into unbounded in-memory channels. Consumer drains them on each poll tick.

HTTP: server-side per-consumer buffer (long-poll or receipt DB). Consumer drains it with the same call.

🔍 On-demand queries

All ILifeCycleRuntimeBus methods

The consumer initiates these. No event was fired. Examples: fetch a timeline to email it, list pending ACKs for a dashboard, upsert a runtime status log from a wrapper, suspend/resume/reopen an instance.

In-process: direct call to engine. HTTP: POST/GET to engine's REST endpoint. Consumer never knows the difference.

Common confusion
ListPendingAcksAsync (on-demand query) is NOT the same as GetDueTransitionsAsync (event delivery). The former is a consumer asking "what's unresolved in the engine DB right now?". The latter is the engine's events arriving at the consumer through the proxy's buffering mechanism.
